# Agent Workflow Reference

Quick reference for the development loop used in this repository.

## The Loop

```
Define → Plan → Task → Assign → Implement → Review → Merge → Document
```

### 1. Define

Update `docs/product.md` with requirements before work begins.

Questions to answer:

- What problem does this solve?
- Who is the user?
- What does done look like?
- What is explicitly out of scope?

### 2. Plan

Update `docs/roadmap.md` with the current milestone and upcoming work.

- Break goals into milestones
- Identify dependencies and risks
- Keep the roadmap current — stale plans mislead agents

### 3. Create Tasks

Add concrete work items to `TASKS.md`.

Good tasks are:

- Specific and actionable
- Completable in a single session
- Independent or have clear dependencies noted
- Scoped to one area of the codebase

### 4. Assign

Ask an agent to take a task. The agent should:

- Read `docs/product.md` and `docs/architecture.md` first
- Read `TASKS.md` to see what is in progress and blocked
- Move the task to **In Progress** in `TASKS.md`, noting the agent name
- Ask clarifying questions before implementing — do not guess at requirements

**Open questions:** If a task depends on unresolved decisions in `docs/product.md`, `docs/architecture.md`, or `docs/roadmap.md`, flag it immediately. Do not implement work that depends on unanswered questions.

### 5. Implement

The agent implements the task following the engineering principles in [`AGENTS.md`](AGENTS.md):

- Small, focused changes
- No unrelated refactoring
- Tests included
- Implementation notes left in `TASKS.md`

**If you discover a blocker:**

1. Move the task to **Blocked** in `TASKS.md`
2. Write a clear description of what is blocking progress
3. Identify who needs to resolve it (tag the human or owner)
4. Do not proceed with a workaround without explicit approval

**Implementation notes format:**
When moving a task to "Ready For Review", add notes explaining what you did:

```md
- [Agent]: Implement password reset email - [https://github.com/org/repo/pull/123] or [feature/auth-reset-email]
  - Implementation notes:
    - Used [Resend](https://resend.dev) for email delivery (see ADR 002)
    - Added rate limiting: max 5 resets/hour per user
    - Tests in `tests/auth/reset.test.ts`
    - Blocked on: Resend API key in .env (waiting for human setup)
```

### 6. Review

A _different_ agent reviews the implementation:

- Correctness against requirements
- Architecture consistency
- Edge cases and error handling
- Test coverage
- Security considerations

The reviewing agent comments directly on the code or leaves notes in `TASKS.md`.

**If review identifies issues:**

- Reviewer leaves specific comments
- Agent moves task back to **In Progress**
- Agent revises and moves to **Ready For Review** again
- Loop continues until review passes

### 7. Merge

The human reviews the final result and approves.

**Merge checklist:**

- [ ] Requirements fully implemented
- [ ] Tests pass and cover new behavior
- [ ] No breaking changes to architecture
- [ ] `TASKS.md` and `docs/` updated
- [ ] ADR created (if significant decision)
- [ ] Code review completed by different agent

No agent merges its own work.

### 8. Document Decisions

When a significant technical decision is made, create an ADR in `docs/adr/`.

See [`docs/adr/000-template.md`](docs/adr/000-template.md) for the blank template.

## TASKS.md Format

```md
# In Progress

- [Agent]: [Task description]

# Ready For Review

- [Agent]: [Task description]
  - Implementation notes: [brief summary]

# Blocked

- [Task description]
  - Blocked by: [reason]

# Backlog

- [Task description]

# Done

- [Task description] — completed [date]
```

## Definition of Done

A task is complete when:

- [ ] Requirements are implemented
- [ ] Code compiles without errors
- [ ] Relevant tests pass
- [ ] `TASKS.md` is updated
- [ ] Documentation reflects any changes
- [ ] An ADR exists for any significant decisions
- [ ] A different agent has reviewed the code
- [ ] Human has approved

## Cross-Agent Review Pairings

| Author  | Reviewer          |
| ------- | ----------------- |
| Claude  | Codex or Gemini   |
| Codex   | Claude or Copilot |
| Copilot | Gemini or Claude  |
| Gemini  | Codex or Copilot  |

Rotate pairings when possible to reduce systematic blind spots.
